# CodeDiff line-wrapping design

Research snapshot: 2026-07-16
Manual validation status updated: 2026-07-19

## Executive decision

Build aligned multi-pane wrapping for Neovim 0.12+ and complete inline wrapping for Neovim 0.13+. Do not accept permanently independent scrolling or treat a delayed `wrap=true` assignment as a solution.

The proposed root solution combines:

1. CodeDiff's line mappings as semantic alignment anchors.
2. `nvim_win_text_height()` as the source of truth for rendered height.
3. Experimental window-scoped namespaces through `nvim__ns_set()` so each pane can own width-specific filler rows even when panes share a buffer.
4. Neovim 0.13's wrapped virtual lines for complete inline-diff wrapping.
5. Lightweight semantic view synchronization from `WinScrolled`; native `scrollbind` remains disabled in all Neovim 0.12+ multi-pane views because it drifts with wrapping and can oscillate inside large virtual filler blocks.

Recompute alignment after rendering, edits, layout changes, folds, or width changes. Never recompute on ordinary scrolling.

The prototype proved that `nvim_win_text_height()` can drive compensation, but native `scrollbind` still drifts. The working design therefore uses pane-scoped padding plus lightweight custom view synchronization. Dedicated mirror buffers have not been necessary.

## Current rendering and alignment model

CodeDiff computes VSCode-style line mappings and always renders line, character, and move decorations. The policy then diverges:

- `wrap=false` retains logical `virt_lines` fillers and opposite-side move fillers, then uses renderer-aware `WinScrolled` synchronization on Neovim 0.12+ with native `scrollbind` as an older-version fallback.
- Aligned `wrap=true` skips those balancing fillers, builds one n-ary semantic plan, measures settled display heights, adds window-scoped compensation, and uses the same display-facade synchronization primitives.

Relevant files:

- Diff orchestration: `./lua/codediff/ui/view/render.lua`
- Logical filler calculation and rendering: `./lua/codediff/ui/core.lua`
- Pure n-ary plan: `./lua/codediff/ui/wrap_alignment_model.lua`
- Alignment coordinator and invalidation: `./lua/codediff/ui/wrap_alignment.lua`
- Measurement and compensation: `./lua/codediff/ui/wrap_alignment_renderer.lua`
- Opaque viewport capture and synchronization: `./lua/codediff/nvim/display/viewport.lua` and `events.lua`
- Window-scoped padding ownership: `./lua/codediff/nvim/display/layer.lua`
- Renderer-backed measurement and fold queries: `./lua/codediff/nvim/display/layout.lua`
- Merge rendering: `./lua/codediff/ui/core.lua`
- Move annotations and unwrapped-mode fillers: `./lua/codediff/ui/move.lua`
- Side-by-side window creation: `./lua/codediff/ui/view/side_by_side.lua`
- Inline rendering: `./lua/codediff/ui/inline.lua`
- Inline window policy: `./lua/codediff/ui/view/inline_view.lua`
- Conflict result window: `./lua/codediff/ui/view/conflict_window.lua`

- Auto-refresh view restoration: `./lua/codediff/ui/auto_refresh.lua`
- Resize handling: `./lua/codediff/ui/lifecycle/cleanup.lua`

The retained unwrapped fillers align logical buffer rows, not wrapped screen rows. Enabling wrapping without the measured compensation pass would therefore make all later lines drift when corresponding lines consume unequal display heights.

Example at a 30-column text width:

```text
original line 3: LLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLLL
modified line 3: short

original height: 2 display rows
modified height: 1 display row
```

No current filler is generated because both sides contain one logical line. Every subsequent corresponding line is displaced by one screen row.

### Minimal reproduction

1. Create two files containing identical sentinel lines around one changed line.
2. Make the changed line long on one side and short on the other.
3. Open the files side by side in CodeDiff.
4. Enable `wrap` in both panes.
5. Narrow the panes and scroll below the changed line.

Wrapping both panes is insufficient. Correctness requires equal cumulative display height at every semantic alignment boundary.

The former downstream delayed `wrap=true` workaround in `/home/rich/code/nix-private/out-of-store-config/nvim/plugin/60_git-diff_codediff.lua` has been removed. The local configuration now enables CodeDiff's aligned `diff.wrap` implementation directly.

## Required invariants

For every corresponding alignment boundary:

```text
display_rows(original prefix)
  == display_rows(modified prefix)
  == display_rows(result prefix, when applicable)
```

Display height includes:

- Wrapped real lines.
- Logical diff fillers.
- Wrap-compensation fillers.
- Move annotations.
- Compact folds.
- Inline and end-of-line virtual text.
- Signs, numbers, fold columns, status columns, and actual pane width.
- `linebreak`, `breakindent`, `showbreak`, tabs, Unicode width, conceal, and other display-affecting behavior.

Operational invariants:

- Measure after final layout and window options have been applied.
- Use each pane's actual width; do not assume equal pane widths.
- Activate wrapped-view synchronization only after padding is current.
- Keep cursors independent; CodeDiff should not introduce `cursorbind`.
- Preserve semantic position and `topline`, `topfill`, and `skipcol` where applicable.
- Rebuild after edit, refresh, resize, layout, fold, file-selection, and resume changes.
- Never use delayed timers to reapply wrapping.
- Never perform full-document measurement on `WinScrolled`, `CursorMoved`, or each scroll input.

### Neovim 0.11

Neovim 0.11 fixed `scrollbind` accounting for virtual lines. Screen-line offsets now include `virt_lines`:

- Issue: https://github.com/neovim/neovim/issues/29751
- Fix: https://github.com/neovim/neovim/pull/29766
- Release notes: https://neovim.io/doc/user/news-0.11/

This means the remaining side-by-side problem is not simply that `scrollbind` ignores CodeDiff's fillers. CodeDiff must add enough fillers to compensate for unequal wrapped heights. The later manual prototype also showed that native `scrollbind` still drifts after compensation, so wrapped mode requires custom lightweight view synchronization.

### Neovim 0.13

Neovim 0.13 has merged wrapped virtual-line overflow:

- PR: https://github.com/neovim/neovim/pull/40113
- New values: `virt_lines_overflow = "wrap" | "auto"`

This materially improves inline CodeDiff. Deleted/original lines rendered as `virt_lines` can wrap instead of truncating. `"auto"` should wrap in wrapped windows while retaining appropriate overflow behavior in unwrapped windows.

It does not solve side-by-side alignment automatically.

### Window-scoped namespaces

The experimental `nvim__ns_set()` API can scope a namespace to specific windows:

```lua
local ns = vim.api.nvim_create_namespace("CodeDiffAlignment:" .. win)
vim.api.nvim__ns_set(ns, { wins = { win } })
```

References:

- Initial implementation: https://github.com/neovim/neovim/pull/28728
- Stabilization request: https://github.com/neovim/neovim/issues/36994

Padding extmarks remain associated with a buffer, but rendering is restricted to their pane. This may solve the mismatch between buffer-local extmarks and window-local wrapping, including two differently sized CodeDiff windows showing the same buffer.

The API is experimental and may change. All use must be isolated behind a small adapter.

### Measurement APIs

`nvim_win_text_height()` is the preferred measurement primitive because it:

- Measures the rendering of a specific window.
- Works for off-screen ranges.
- Accounts for wrapping and folds.
- Separately reports virtual lines and native diff fillers.
- Supports bounded measurement.

Documentation: https://neovim.io/doc/user/api/#nvim_win_text_height()

The alternatives are insufficient as the primary model:

- `screenpos()` only measures visible positions.
- `winline()` is viewport-dependent.
- `virtcol()` measures horizontal cells, not rendered row count, and does not model all conceal behavior.
- `strdisplaywidth()` cannot reproduce first-row and continuation widths, gutters, wrapping options, folds, conceal, or arbitrary decorations.

### Native diff mode

Native `:diffthis` is not a root solution. Neovim documents that diff alignment can fail when corresponding lines wrap to different numbers of screen rows:

- Documentation: https://neovim.io/doc/user/diff/#view-diffs
- Open issue: https://github.com/neovim/neovim/issues/29518

CodeDiff would also lose control over filler placement and take on global `diffexpr`/`diffopt` integration concerns.

### Folds

Neovim still has no general fold-change event:

- Request: https://github.com/neovim/neovim/issues/29870
- Closed prior attempt: https://github.com/neovim/neovim/pull/24279

CodeDiff must therefore own or synchronize compact-fold operations, or provide another cheap invalidation mechanism.

## Neovim API and ecosystem follow-up

Research updated 2026-07-18 using `gh` issue, pull-request, and code search plus Neovim master at commit [`952e59347e05`](https://github.com/neovim/neovim/commit/952e59347e053a61cb621e56625c846b249236a0) in `~/code/reference-repos/neovim`.

### What already exists in Neovim

#### `nvim_win_text_height({ max_height = ... })` is a partial inverse

PR [#32835](https://github.com/neovim/neovim/pull/32835) added `max_height` to `nvim_win_text_height()`. In addition to limiting work, it returns `end_row` and `end_vcol` where the requested rendered height is reached. This is the closest current public API to rendered-row-offset-to-buffer-position conversion.

Neovim itself uses this pattern in its UI2 message implementation:

```lua
local texth = vim.api.nvim_win_text_height(win, { max_height = height })
local row = texth.end_row
local col = vim.fn.virtcol2col(win, row + 1, texth.end_vcol)
local screen_col = vim.fn.screenpos(win, row + 1, col).endcol
```

Source: [`runtime/lua/vim/_core/ui2/messages.lua`](https://github.com/neovim/neovim/blob/952e59347e053a61cb621e56625c846b249236a0/runtime/lua/vim/_core/ui2/messages.lua#L118-L140).

This pattern now provides the guarded fast path in `./lua/codediff/nvim/display/viewport.lua`, replacing the former synchronizer's prefix search for proven text endpoints while retaining measured fallback searches for ambiguous cases:

- The prefix search for the buffer row containing an absolute rendered offset.
- The virtual-column search used to reconstruct `skipcol` inside a wrapped line.

It is not a complete inverse. The return type cannot identify whether the requested height lands in text, a fold, or a filler row, nor can it return the offset within a filler block. A Neovim 0.12.2 probe targeting `virt_lines_above` produced negative `end_vcol` values, demonstrating that filler endpoints cannot be consumed blindly. A robust CodeDiff path must identify filler boundaries and retain a fallback for ambiguous results.

A hybrid lookup should require approximately one to three `nvim_win_text_height()` calls per target pane instead of O(log lines + log virtual columns) calls:

```text
absolute rendered offset
→ max_height lookup
→ classify endpoint as text/fold/filler
→ text: restore topline + skipcol
→ filler: restore topline + topfill
→ ambiguous: use the existing measured fallback
```

Issue [#16166](https://github.com/neovim/neovim/issues/16166) explicitly requests an extended scrolling API aware of filler lines and `skipcol`. The older viewport API attempt [#15248](https://github.com/neovim/neovim/pull/15248) was closed; maintainers linked it to the broader API cleanup proposal rather than replacing it with a rendered-row setter. Issue [#11370](https://github.com/neovim/neovim/issues/11370) records `winrestview()` as the current low-level workaround.

#### Settled viewport machinery exists internally

Neovim's `ui_ext_win_viewport()` already computes the information needed for renderer-aware scrolling. It:

- Delays `win_viewport` until pending redraw updates are ready to flush.
- Computes `scroll_delta` with the same internal `win_text_height()` primitive.
- Accounts for folds, virtual/filler rows, `topfill`, and `skipcol`.
- Stores the prior viewport fields per window.

Source: [`src/nvim/window.c`](https://github.com/neovim/neovim/blob/952e59347e053a61cb621e56625c846b249236a0/src/nvim/window.c#L985-L1044). PR [#19270](https://github.com/neovim/neovim/pull/19270) added rendered `scroll_delta`; PR [#24182](https://github.com/neovim/neovim/pull/24182) delayed the event until screen updates are sent.

This explains CodeDiff's observed timing issue. In the normal-mode loop, `WinScrolled` runs before `normal_redraw()`, even though topline and cursor validation have occurred. Source: [`src/nvim/normal.c`](https://github.com/neovim/neovim/blob/952e59347e053a61cb621e56625c846b249236a0/src/nvim/normal.c#L1428-L1462). A keyboard scroll can therefore settle to a different wrapped-row state during the following redraw.

The internal event is not currently usable by an in-process plugin:

- `win_viewport` is an external UI/multigrid event.
- Experimental `vim.ui_attach()` accepts global externalized widgets but not `ext_multigrid`, so Lua cannot subscribe to `win_viewport` through it.
- Issue [#8539](https://github.com/neovim/neovim/issues/8539) was closed when the external UI event landed, but that did not provide a plugin-facing window event stream.
- A raw callback during UI flush might still need to schedule mutations for redraw safety. A proper plugin API should be a post-redraw event or window attachment callback, not merely exposure of an unsafe internal callback.

Exposing only the existing event would be a meaningful but incomplete improvement. CodeDiff could remove the immediate pre-redraw synchronization, source-window heuristics, and some source-side measurement. Existing `scroll_delta` is relative and may be approximate for jumps larger than a screen, and the payload does not include an exact absolute rendered offset, `topfill`, or `skipcol`. Target panes would still require rendered-offset-to-view conversion.

The ideal API shape is:

```lua
vim.api.nvim_win_attach(win, {
  on_viewport = function(_, state)
    -- state.display_offset: exact absolute rendered-row offset
    -- state.scroll_delta: exact rendered-row delta
    -- state.topline, state.topfill, state.skipcol
    -- callback runs after the viewport has settled
  end,
})

vim.api.nvim_win_set_viewport(win, {
  display_offset = offset,
})
```

A settled event alone could remove roughly one-third to one-half of the compatibility logic now isolated in `./lua/codediff/nvim/display/events.lua`. An exact absolute offset plus a filler-aware setter could remove approximately 80–90% of `viewport.lua` and `events.lua`, including fallback searches and most scheduling/suppression machinery. It would not remove semantic alignment planning, measured padding, namespace ownership, fold invalidation, or display invalidation.

#### Native `scrollbind` still uses logical/filler rows, not wrapped rows

Current `scrollbind` computes its vertical position with `get_vtopline()`, which calls `plines_m_win_fill()`. That helper explicitly does not count a wrapped physical line as multiple screen rows. Sources:

- [`src/nvim/normal.c`](https://github.com/neovim/neovim/blob/952e59347e053a61cb621e56625c846b249236a0/src/nvim/normal.c#L2102-L2200)
- [`src/nvim/plines.c`](https://github.com/neovim/neovim/blob/952e59347e053a61cb621e56625c846b249236a0/src/nvim/plines.c#L934-L950)

PR [#29766](https://github.com/neovim/neovim/pull/29766) added virtual and diff-filler accounting but did not change this wrapped-row limitation. Issue [#29518](https://github.com/neovim/neovim/issues/29518) remains open. This confirms from source that native `scrollbind` cannot replace custom wrapped synchronization even after CodeDiff installs compensation rows.

#### Window-scoped namespaces are established but still experimental

PR [#28728](https://github.com/neovim/neovim/pull/28728) fully implemented namespace scoping through `nvim__ns_set(ns, { wins = {...} })`, including redraw and window lifecycle handling. Stabilization issue [#36994](https://github.com/neovim/neovim/issues/36994) remains open.

Neovim's own runtime now uses this API to scope operation highlights to a window in `vim.hl`: [`runtime/lua/vim/hl.lua`](https://github.com/neovim/neovim/blob/952e59347e053a61cb621e56625c846b249236a0/runtime/lua/vim/hl.lua#L199-L214). This does not make the API stable, but it confirms that CodeDiff's isolated adapter follows an upstream-used pattern rather than an untested private mechanism.

#### Display and fold invalidation remain missing

Neovim has internal global and per-window `display_tick` fields, but they identify redraw rounds; they are not a public generation that changes only when rendered text height changes. The internal `w_viewport_invalid` flag similarly drives UI viewport emission rather than exposing decoration/layout invalidation to plugins.

No broad event or generation was found for height-affecting changes across extmarks, virtual text/lines, signs, conceal, inlay hints, folds, dynamic gutters, and status-column providers. Open issue [#32185](https://github.com/neovim/neovim/issues/32185) proposes only `SignChanged`.

Fold support remains fragmented:

- [#29870](https://github.com/neovim/neovim/issues/29870) requests `FoldOpen` and `FoldClose` autocmds.
- [#8538](https://github.com/neovim/neovim/issues/8538) requests fold UI events.
- [#19226](https://github.com/neovim/neovim/issues/19226) proposes fold CRUD, queries, and attachment callbacks.
- PR [#24279](https://github.com/neovim/neovim/pull/24279) implemented `FoldChanged` but closed unmerged.
- Draft PR [#31559](https://github.com/neovim/neovim/pull/31559) adds fold CRUD/query APIs but no events.

No batch `nvim_win_text_height()` API or rendered-layout API was found. CodeDiff must continue issuing per-range measurements and maintaining conservative caches.

### Improvements implemented without new Neovim APIs

The first two research recommendations are complete in the display facade. The remaining items are optional follow-up or retained boundaries.

#### 1. Guarded `max_height` target lookup — completed

`./lua/codediff/nvim/display/viewport.lua` implements this high-level flow:

```text
capture source absolute rendered offset
→ resolve each target offset with max_height
→ restore target view
→ preserve target cursor
```

The endpoint resolver:

- Special-case offset zero and end-of-buffer clamping.
- Use `end_row`/`end_vcol` directly only for a proven text endpoint.
- Measure the endpoint's filler boundary to derive `topfill` when the target lands in compensation or other virtual rows.
- Treat negative or otherwise inconsistent `end_vcol` as ambiguous.
- Retains the measured search for ambiguous endpoints, protected by focused direct/fallback equivalence tests.

Observed benefit: a representative direct lookup used 7 text-height calls versus 22 through the forced fallback while preserving Neovim's renderer as the source of truth.

#### 2. Synchronize only from the scheduled settled view — completed

`display/events.lua` coalesces the early `WinScrolled` signal and performs one synchronization from the settled source view:

```text
WinScrolled
→ identify and coalesce source pane
→ next event-loop turn
→ capture final source view once
→ synchronize partners once
```

Manual keyboard and inactive-pane mouse testing confirmed aligned settled offsets without alignment rebuilds. Generated partner events are suppressed inside the facade.

#### 3. Consider a redraw observer only as a follow-up experiment

The `vscode-neovim` decoration-provider pattern can observe `winsaveview()` during redraw and may provide a more direct settled-view signal. It is viable as a prototype but should not replace the simpler scheduled `WinScrolled` path without evidence because:

- Decoration-provider callbacks have redraw-context restrictions.
- Synchronizing partner windows must still be scheduled.
- The callback runs on unrelated redraws and requires view caching and source/target suppression.
- It does not solve offset-to-target conversion.

#### 4. Retain current namespace and explicit invalidation boundaries

The extracted `nvim/display/layer.lua` facade is the isolation boundary for `nvim__ns_set()`. Copying another plugin's wrapper would not simplify ownership across shared buffers and sessions.

Do not adopt fold-key interception as the primary invalidation mechanism. It can be an optional convenience but cannot replace CodeDiff-owned fold invalidation and the explicit `invalidate_alignment()` API.

Acceptance for the first two improvements should include:

- `tests/ui/wrap_smoothscroll_spec.lua` on unequal two- and three-pane widths.
- Targets inside wrapped text, compensation rows, move-leading rows, folds, BOF, and EOF.
- Neovim 0.12.2 and the pinned 0.13 nightly.
- Repeated `<C-e>/<C-y>`, `<C-d>/<C-u>`, Page Up/Down, and mouse wheels from every pane.
- API-call and elapsed-time comparison before and after the target resolver change.
- Confirmation that scrolling still does not increment the alignment rebuild counter.

### Isolation plan: an internal ideal display API

The Neovim display workarounds should not remain distributed through CodeDiff's view, alignment, renderer, and lifecycle modules. Introduce one internal facade that models the APIs CodeDiff wishes Neovim exposed. CodeDiff's main code should consume that contract without knowing how `topfill`, `skipcol`, early `WinScrolled`, experimental namespace scoping, or rendered-position searches work.

Do not monkeypatch `vim.api` or name these functions as if they were public Neovim APIs. The facade is a private CodeDiff contract whose implementation can be replaced when Neovim gains equivalent primitives.

#### Adopted external techniques

The implementation adapts proven techniques found in Neovim and active plugins rather than inventing another independent model:

1. **Use Neovim's own bounded inverse pattern.** `display.viewport` uses `nvim_win_text_height({ max_height = ... })`, `end_row`, and `end_vcol`, following Neovim UI2's implementation in [`runtime/lua/vim/_core/ui2/messages.lua`](https://github.com/neovim/neovim/blob/952e59347e053a61cb621e56625c846b249236a0/runtime/lua/vim/_core/ui2/messages.lua#L118-L140), with a guarded fallback for endpoints the current return type cannot represent.

2. **Use the simplest proven settled-view observation.** `display.events` uses scheduled, coalesced `WinScrolled` handling. The decoration-provider technique from [`vscode-neovim/runtime/lua/vscode/viewport.lua`](https://github.com/vscode-neovim/vscode-neovim/blob/5a2abebb5bf5e76afb05faa98b479ece5b7e07ac/runtime/lua/vscode/viewport.lua#L21-L66) remains an optional experiment only if evidence shows the scheduled path is insufficient.

3. **Use renderer-backed range counting and characterize fallbacks.** Follow [`nvim-scrollview`'s `proper_line_count()` and `proper_virtual_topline_lookup()`](https://github.com/dstein64/nvim-scrollview/blob/0efc4e82094dbda2e411f7c8d44928716d00b655/lua/scrollview.lua#L833-L1120) in preferring `nvim_win_text_height()` over arithmetic estimation and in retaining explicit fallback behavior. Do not copy its proportional scrollbar lookup as an exact viewport mapping.

4. **Account explicitly for filler at view boundaries.** Preserve the useful `topfill` boundary accounting demonstrated by [`Snacks.scroll`'s `scroll_lines()`](https://github.com/folke/snacks.nvim/blob/882c996cf28183f4d63640de0b4c02ec886d01f2/lua/snacks/scroll.lua#L239-L263), while extending it with CodeDiff's required width-specific `skipcol` handling.

5. **Keep per-window namespace ownership explicit.** Model `display.layer` after the lifecycle pattern in [`gitsigns.nvim`'s deleted preview](https://github.com/lewis6991/gitsigns.nvim/blob/31d6fb2d618bca1482b9f274751ead5f03461408/lua/gitsigns/deleted_preview.lua#L184-L229): one scoped namespace per window, clear old-buffer marks before reuse, clear scope during teardown, and retain capability detection. Preserve CodeDiff's stronger shared-session ownership and extmark-reuse behavior.

6. **Reuse measurement and lookup caches conservatively.** The exported helpers in [`satellite.nvim`](https://github.com/lewis6991/satellite.nvim/blob/87843c9c8f28b54332497302de380a6d94c9e82b/lua/satellite/util.lua#L24-L157) demonstrate per-window rendered-height caching. Keep CodeDiff's stricter content/signature/decorated-row eligibility. Do not adopt Satellite's [`vim.on_key()` fold interception](https://github.com/lewis6991/satellite.nvim/blob/87843c9c8f28b54332497302de380a6d94c9e82b/lua/satellite.lua#L102-L161) as a correctness mechanism.

7. **Use bounded rendered measurements around scrolling.** [`go-up.nvim`](https://github.com/aidancz/go-up.nvim/blob/5e325fed234232c8fe389f5c2637f6fd91ce6d91/lua/go-up.lua#L155-L263) demonstrates using `max_height` to bound work for interactive scrolling. Apply the bounded-measurement idea without adopting its normal-command and cursor-driven viewport control.

8. **Do not import plugin-specific mirror geometry.** [`vimade`](https://github.com/TaDaa/vimade/blob/3d3d2db7ecd43c0181b20fede11d26f090dbc0d9/lua/vimade/focus/core.lua#L394-L558) provides useful evidence for combining text heights with `topfill`/`skipcol`, but its floating mirror windows and deliberate `topfill=0` restoration are not a reusable absolute-offset API.

These are source references, not proposed runtime dependencies. Copy only the narrow ideas that improve the facade, retain attribution where source is materially copied, and keep CodeDiff's tests authoritative.

- Two differently sized windows sharing one buffer.
- An ordinary same-buffer window with no decorations.
- Re-scope, buffer replacement, extmark reuse/update/removal, and teardown.
- Concurrent independent layers and capability failure.

CodeDiff-level wrapping tests should then focus on orchestration: selecting pane groups, building semantic plans, calculating padding, requesting rebuilds, and preserving session ownership. They should not re-test raw `topfill`/`skipcol` mechanics.

Run the display tests against Neovim 0.12.2 and the pinned 0.13 nightly. The lower-level suite becomes the compatibility contract for replacing hacks with native APIs later.

#### Migration sequence

Completed in order; the architecture spec now enforces the resulting boundary.

1. **Characterize and document the current behavior.** Create `nvim/display/README.md` with the initial hack inventory and add lower-level viewport, event, layout, and layer tests around the existing implementations before moving code.
2. **Extract viewport operations without behavior changes.** Move capture, restore, offset measurement, and target lookup into `display.viewport`; keep the current searches initially.
3. **Improve the isolated viewport implementation.** Add the guarded `max_height` resolver and remove the immediate pre-settlement synchronization so only one coalesced settled sync runs.
4. **Extract event normalization.** Move scroll settling, target-event suppression, option/diagnostic/conceal observation, and subscription cleanup into `display.events`.
5. **Extract window-local layers.** Move namespace capability/scoping and padding extmark lifecycle into `display.layer`.
6. **Extract layout inspection.** Move range measurement, signatures, decorated rows, and fold queries into `display.layout` while leaving semantic interval merging and cache policy in the renderer.
7. **Delete superseded modules and fields.** Remove `wrap_view_sync.lua`, `window_namespace.lua`, `group.ignore_wins`, and raw display-event handlers from main wrapping code.
8. **Enforce the architecture.** Add a lightweight check that `nvim__ns_set` and `nvim_win_text_height` occur only under `./lua/codediff/nvim/display/`, and that wrapping modules outside the facade do not reference `topfill`, `skipcol`, or `WinScrolled`.
9. **Rerun full acceptance.** Run focused display tests, all wrapping specs, the full Lua suite, 1k/10k/100k benchmarks, and the recorded two-/three-pane manual matrix on both target Neovim versions.

Do not add speculative `native` and `compat` backends now. There is one implementation of one internal contract. When Neovim exposes a settled viewport event, display offset setter, stable scoped namespaces, display generation, fold event, or batch measurement API, replace only the corresponding file inside `nvim/display/`.

## Evaluated solution families

### Wrap with `scrollbind` disabled

Advantages:

- Small implementation.
- Makes long prose readable.
- No false claim of alignment.

Disadvantages:

- Panes drift immediately during independent scrolling.
- Hunk navigation must reposition every pane explicitly.
- Poor conflict/result comparison experience.

This remains a possible degraded fallback, not the complete feature.

### Native Neovim diff mode

Advantages:

- Familiar Vim behavior.
- Native logical fillers.

Disadvantages:

- Still fails on unequal wrapped heights.
- Changes highlighting, folding, options, and integration semantics.
- `diffexpr` and `diffopt` are global.
- Does not naturally preserve CodeDiff's move and merge rendering.

Do not pursue this as the root architecture.

### Arithmetic wrap estimation

Earlier CodeDiff experiments used `strdisplaywidth()` and calculated expected wraps arithmetically.

Disadvantages:

- Cannot faithfully model the complete renderer.
- Easily diverges on tabs, Unicode, conceal, gutters, continuation indentation, folds, and virtual text.
- Requires duplicating Neovim display logic.

Use Neovim's renderer as the source of truth instead.

### Display-row compensation with shared buffer namespaces

Advantages:

- Can align normal two-pane sessions.
- Works with CodeDiff's existing rendering model.

Disadvantages:

- Padding depends on window width but ordinary extmarks affect every window showing the buffer.
- Multiple sessions at different widths can conflict or leak fillers into normal windows.

This is acceptable only with a one-owner limitation. Window-scoped namespaces are the preferred replacement.

### Display-row compensation with window-scoped namespaces

Advantages:

- Each pane owns width-specific filler rows.
- Multiple windows can display the same buffer at different widths.
- Retains CodeDiff rendering and native scrolling.
- Avoids editable mirror-buffer synchronization.

Risks:

- Depends on an experimental API.
- Must prove measurement and `scrollbind` honor namespace scoping.
- Namespace/window ownership must be updated on layout replacement and cleanup.

This is the recommended future-facing architecture.

### Inline wrapping

On Neovim 0.12 and earlier, deleted/original `virt_lines` cannot wrap. A window can wrap real modified lines, but the old side remains truncated.

On Neovim 0.13, use `virt_lines_overflow = "auto"` or `"wrap"`. This avoids a projection-buffer rewrite and makes complete inline wrapping feasible.

## Recommended architecture

### Outside-in control flow

The display facade described in the isolation plan is the only wrapping code allowed to manipulate Neovim's rendered viewport or window-scoped padding primitives.

```lua
-- ./lua/codediff/ui/view/render.lua

local display = require("codediff.nvim.display")

local saved_view = display.capture(active_win)
layout.arrange(tabpage)
window_policy.apply(session)

local rendered = renderer.render_base_decorations(...)
local plan = alignment_model.build(diff_result, rendered)

wrap_alignment.apply({
  plan = plan,
  panes = session.diff_panes,
})

display.restore(active_win, saved_view)
wrap_alignment.synchronize_from(tabpage, active_win)
```

Authoritative ordering:

```text
display facade captures an opaque semantic/rendered anchor
→ CodeDiff disables native scrollbind
→ arrange windows
→ configure wrap and stable width-affecting options
→ display facade creates or updates pane-scoped layers
→ display facade clears generated padding
→ CodeDiff renders highlights, logical gaps, and annotations
→ CodeDiff builds semantic alignment intervals
→ display facade measures pane ranges
→ CodeDiff computes n-ary display-row compensation
→ display facade installs window-local padding
→ verify cumulative prefix equality
→ display facade restores the active anchor
→ synchronize partner panes through rendered offsets
→ activate the display facade's settled viewport observer
```

### API shape

The authoritative low-level API is `./lua/codediff/nvim/display/init.lua`, defined in **Isolation plan: an internal ideal display API**. Only files under `./lua/codediff/nvim/display/` may call `nvim__ns_set()`, interpret `topfill`/`skipcol`, or translate rendered offsets with `nvim_win_text_height()`.

CodeDiff's semantic coordinator remains small:

```lua
-- ./lua/codediff/ui/wrap_alignment.lua

---@param opts { plan: table, panes: table[], reason?: string }
---@return table? stats
function M.apply(opts)

---@param tabpage integer
---@param source_win integer
function M.synchronize_from(tabpage, source_win)

---@param tabpage integer
---@param reason "render"|"edit"|"resize"|"fold"|"layout"|"resume"|"external-display"
function M.rebuild(tabpage, reason)

---@param win integer
function M.clear_window(win)
```

The coordinator owns semantic groups and lifecycle policy; the display facade owns every Neovim renderer workaround. The former `window_namespace.lua` and `wrap_view_sync.lua` modules have been removed rather than retained as permanent architectural seams.

### Alignment algorithm

The alignment plan describes corresponding semantic intervals rather than merely storing a list of filler extmarks.

For each interval:

1. Measure its natural display height in every participating pane after logical fillers and annotations exist.
2. Compute:

   ```text
   target_height = max(pane interval heights)
   ```

3. Add `target_height - pane_height` blank virtual rows to each shorter pane at the interval boundary.
4. Continue to the next interval.

For a two-pane comparison:

```text
original: 3 rows
modified: 1 row

padding(original) = 0
padding(modified) = 2
```

For conflict mode:

```text
incoming: 2 rows
current: 4 rows
result: 3 rows

target = 4
padding(incoming) = 2
padding(current) = 0
padding(result) = 1
```

Clear all generated compensation before measuring. Logical fillers and semantic annotations should remain present so their heights are included. Render all compensation only after measurement to avoid feedback between intervals.

Special placement is required for empty intervals and beginning/end-of-file gaps using `virt_lines_above` or a previous-row anchor.

### Alignment anchors

Anchors should come from CodeDiff's actual mappings, including inner changes where CodeDiff currently aligns lines inside a changed block.

The model should represent:

- Corresponding unchanged lines.
- Inserted/deleted empty ranges.
- Multiline replacement boundaries.
- Move annotation rows.
- Folded compact regions.
- Merge-base intervals for conflict panes.

Measuring only hunks is insufficient when pane widths or decorations differ, because unchanged long lines can wrap differently.

A performance optimization may skip known-safe intervals, but correctness must not depend on reimplementing Neovim's wrap calculation.

## Mode-specific integration

### Side by side

Both panes use `wrap=true`, pane-scoped filler namespaces, and measured compensation. Native `scrollbind` is disabled; `WinScrolled` synchronizes panes by absolute rendered-row offset.

### Single-file views

Wrapping is safe without cross-pane alignment. The same window policy should still configure it consistently.

### Explorer and history

Selecting a file rerenders into existing windows. Rebuild alignment after every file selection. Opening or closing panels changes pane width and invalidates all measurements.

Explorer/history panels themselves should remain `nowrap`.

### Layout toggle

Window IDs and widths may change. Update namespace scopes, clear stale ownership, arrange windows, and rebuild from scratch.

### Auto-refresh and edits

Capture the active pane's semantic position, recompute the diff and alignment, restore the active view, align partners, and then reactivate custom wrapped-view synchronization.

Debounce edits, but do not use delayed wrapping as a correctness mechanism.

### Suspend and resume

Persist enough state to reconstruct pane namespace ownership and semantic view. Do not rely only on cursor line/column when wrapping is active.

### Move annotations

`./lua/codediff/ui/move.lua` should emit annotations and alignment metadata, not independently create opposite-side balancing fillers. The unified alignment pass should account for annotation height.

The `gm` flow may temporarily disable `scrollbind`, but must restore the configured wrapping/binding policy rather than assuming `scrollbind=true` and `wrap=false`.

### Compact folds

Folded display height is window-local and must be measured as rendered. Fold topology should become part of alignment-plan invalidation.

Because Neovim lacks a fold-change event, prefer CodeDiff-owned fold operations that update corresponding panes and call `alignment.rebuild()`. Arbitrary external fold commands may require a cheap state check or an explicitly documented limitation.

### Conflict/result

Build an n-ary graph using merge-base coordinates:

```text
base interval
  → incoming line range
  → current line range
  → result line range
```

The current incoming/current diffs already map from the base. Result editing requires retaining or recomputing a base-to-result mapping. Every interval should be padded to the maximum height across all three panes.

The implementation computes merge-base-to-incoming, merge-base-to-current, and merge-base-to-result mappings and builds one n-ary interval plan. When an edited result is structurally closer to an input pane than to the merge base, its coordinates compose through that pane instead of relying on a coarse base projection. Complex overlapping and mixed-source conflicts now pass center and bottom fixtures through insert/replace/delete, accept/discard, undo/redo, unequal widths, and scrolling from every pane.

Result accept/discard operations, manual edits, undo, auto-refresh, and resolution must invalidate alignment before restoring views.

### Inline

On Neovim 0.13:

```lua
virt_lines_overflow = "auto"
```

Use wrapping for both real modified lines and deleted/original virtual lines. Verify that the new wrapped virtual rows interact correctly with view restoration and smooth scrolling.

## View synchronization

`winsaveview()` records useful fields, including `topline`, `topfill`, `leftcol`, and `skipcol`, but replaying an old dictionary after filler changes may not preserve the same semantic screen position.

Wrapped mode should capture:

- Active pane and cursor.
- Semantic alignment anchor nearest the top or cursor.
- Display-row offset within that interval.
- `skipcol` for a partially visible wrapped top line when smooth scrolling is supported.

After rebuilding:

1. Restore the active pane against the semantic anchor.
2. Position partner panes at their corresponding anchor.
3. Apply the shared display-row offset.
4. Leave native `scrollbind` disabled and activate semantic view synchronization from that known-aligned baseline.

Cursor positions remain independent. Hunk navigation explicitly moves or reveals partner panes but does not enable `cursorbind`.

Aligned views now preserve smooth scrolling when any owned pane profile enables it. `skipcol` is converted to skipped display rows with `nvim_win_text_height()`, then reconstructed at the target width; offsets that land in compensation rows use `topfill` instead.

## Invalidation model

Rebuild alignment for:

- Initial render.
- Buffer edits after debounce.
- Auto-refresh and external file changes.
- Explorer/history file selection.
- Layout toggle.
- Explorer open/close.
- Window or terminal resize.
- Compact fold changes and fold-option updates.
- Display option, diagnostic, and conceal-mode changes.
- Explicit external decoration or manual-fold invalidation.
- Move annotation changes.
- Conflict/result operations.
- Suspend/resume when window ownership or rendering changed.

Do not rebuild for:

- Mouse-wheel scrolling.
- `<C-e>`, `<C-y>`, `<C-d>`, or `<C-u>`.
- Cursor movement alone.
- Normal pane focus changes.

Cache keys may include:

- Buffer changed tick.
- Window width and text width.
- Relevant display options.
- Fold state generation.
- Decoration generation.
- Alignment-plan generation.

External plugins can add decorations without a general notification. CodeDiff observes stable built-in events and exposes `require("codediff").invalidate_alignment(tabpage, reason)` for manual folds, inlay hints, virtual text/lines, signs, or decoration providers without an observable update event.

### Built-in display and viewport risks

- **Dynamic gutters:** `signcolumn`, `foldcolumn`, `numberwidth`, and `statuscolumn` option changes rebuild automatically. `DiagnosticChanged` catches diagnostic signs and virtual text after update. Providers that mutate signs or dynamic status-column content without changing an option require explicit invalidation.
- **Folds:** compact operations and fold-option changes rebuild automatically. Arbitrary `zo`, `zc`, `za`, and external fold commands align after `invalidate_alignment(tabpage, "fold")`; this explicit call remains necessary because Neovim has no general fold-change event.
- **Dynamic display state:** conceal option/mode changes and diagnostics rebuild automatically. Inline virtual text, virtual lines, enabled inlay hints, and unobservable decoration providers require explicit invalidation.
- **Viewport and layout policy:** aligned wrapping temporarily disables and restores `cursorbind`; resize and owned panel/layout changes rebuild. `scrolloff`, `splitkeep`, `winfixwidth`, and externally managed winbars remain documented integration limits.
- **Renderer inputs:** `linebreak`, `breakindent`, `showbreak`, tabs, `listchars`, columns, conceal, `ambiwidth`, and `display` are included in signatures and option invalidation.

CodeDiff must not respond by rebuilding on every `CursorMoved` or scroll event. It should own or stabilize critical options, rebuild on bounded events such as `OptionSet`, resize, fold actions, and known decoration updates, and document cases that cannot be observed reliably.

## Prototype implementation findings

The Neovim 0.12.2 prototype validated display-row measurement and compensation. A fixture with long/short replacements, insertions, multiline changes, tabs, Unicode, and unequal pane widths produced matching `screenpos()` rows at every visible sentinel after padding.

Native `scrollbind` did not preserve those alignments once scrolling began. Even with equal cumulative display heights and virtual fillers present, `<C-d>` and repeated `<C-e>` produced different `topline` values and visible drift. Calling `:syncbind` did not repair it. This confirms that the 0.11 virtual-line fix is not sufficient for wrapped CodeDiff panes.

The working implementation therefore disables native `scrollbind` in all supported multi-pane views and performs lightweight view synchronization from `WinScrolled`. This was broadened after a Neovim 0.12.2 and 0.13 nightly reproduction showed the same native failure around a 60-row virtual filler block:

1. Identify the pane the user scrolled.
2. Measure its absolute display-row offset from the start of the rendered stream.
3. Restore every partner pane to the same measured offset using prefix-height lookup and `topfill`.
4. Ignore the resulting partner scroll events.

This does not recompute alignment padding while scrolling. Manual tmux runs kept sentinels aligned after `<C-d>`, repeated `<C-e>`, upward scrolling from the opposite pane, hunk navigation, edits with auto-refresh, and pane resizing.

The prototype now covers:

- Normal side-by-side views on Neovim 0.12.2 and 0.13 nightly.
- Neovim 0.13 inline wrapping using `virt_lines_overflow = "auto"`.
- Side-by-side → inline → side-by-side layout transitions without replacing session buffers.
- Compact mode, including synchronized fold opening and closing.
- Move annotations and moved blocks.
- Three-pane conflict views in center and bottom result layouts.
- Result-buffer edits followed by alignment rebuilding.
- Explorer file selection, panel hide/show width changes, and side-by-side rendering.
- Auto-refresh, hunk navigation, resize, and suspend/resume.
- Existing `wrap=false` rendering with logical fillers and renderer-aware synchronization.
- Two concurrent alignment groups and two lifecycle-managed real sessions sharing buffers, plus an ordinary unpadded window.
- 1k-, 10k-, and 100k-line headless performance baselines.

History mode and directory explorer were exercised directly on 0.12 with nested modified files, panel hide/show, unequal pane resizing, and keyboard scrolling from both panes. Git explorer and history also switched through added and deleted single-file previews before returning to an aligned modified file.

Window-scoped namespaces rendered pane-specific fillers successfully. A three-window follow-up showed that `nvim_win_text_height()` honors the namespace window filter on Neovim 0.12.2, so foreign-padding subtraction was removed because it would undercount shared-buffer sessions. Two real CodeDiff sessions reusing both file buffers now pass tab suspend/resume, side/inline/side replacement, resize/rerender, ordinary-window isolation, and cleanup of either session without disturbing the other.

### Current implementation shape

The uncommitted production candidate separates semantic CodeDiff behavior from every Neovim renderer workaround:

- `./lua/codediff/ui/wrap_alignment_model.lua` builds one n-ary plan from merge-base mappings, including inner line mappings, move boundaries, and optional source-composed result coordinates.
- `./lua/codediff/nvim/display/viewport.lua` owns opaque anchors and rendered-offset conversion, including guarded bounded lookup and filler-aware fallback.
- `./lua/codediff/nvim/display/events.lua` owns settled scrolling, common-offset reconciliation, suppression, and normalized display invalidation.
- `./lua/codediff/nvim/display/layout.lua` owns renderer-backed range measurement, opaque rendering contexts, decoration scans, and fold queries.
- `./lua/codediff/nvim/display/layer.lua` owns experimental window scoping and declarative padding-extmark lifecycle.
- `./lua/codediff/ui/wrap_alignment.lua` owns policy, semantic group lifecycle, invalidation, and two-/three-pane orchestration through the facade.
- `./lua/codediff/ui/wrap_alignment_renderer.lua` owns fold-topology collapse, cache policy, height comparison, and semantic padding decisions through the facade.
- `./lua/codediff/ui/wrap_view_sync.lua` and `./lua/codediff/nvim/window_namespace.lua` have been removed.
- Conflict inputs align through merge-base mappings. Each result interval uses its direct base mapping unless an input preserves that region unchanged; exact input mappings are selected per region, with stable source continuity and explicit transition gaps.
- Move annotations are measured as first-class leading display rows instead of creating independent balancing fillers in wrapped mode.

The renderer skips the existing logical filler algorithm when aligned wrapping is enabled; measured compensation covers both logical gaps and unequal wraps. `wrap=false` retains the existing filler path but uses renderer-aware synchronization on Neovim 0.12+. The redraw-sensitive prefix correction and scheduled second initial rebuild have been removed; standard, move, compact, and center/bottom conflict fixtures remained aligned in the single-pass flow.

The authoritative wrapped render order is now:

```text
capture pane views
→ render synchronous diff, move, fold, result, and sign decorations
→ disable native scrollbind and apply final wrap policy
→ redraw once
→ measure every semantic interval
→ install pane-scoped padding
→ restore captured views
→ synchronize partner panes from the active pane
```

`finish_two_pane()` and `finish_conflict()` own the final measure/install/restore/synchronize phase. Ordinary auto-refresh captures before compact-fold refresh and defers the fold module's independent rebuild, preventing measurement against stale decorations. Compact enable/disable/reapply captures before changing fold state when it owns the rebuild. Result refresh renders result highlights and conflict signs before finalizing all three panes. Enabling result auto-refresh no longer schedules an initial render after conflict setup has already rendered and aligned the view.

Semantic-token requests remain asynchronous by design. Their callback only installs highlight extmarks, so waiting for LSP latency would delay the view without changing text height. Future semantic-token changes that add conceal or virtual text would invalidate this assumption and require explicit alignment invalidation.

The current layout toggle directly re-renders existing session buffers for standalone sessions. Explorer and history sessions then force an authoritative rerender of the current selection, preventing stale placeholder buffers from reaching hunk operations. Hidden original buffers retain and later restore their `bufhidden` policy. The complete layout-toggle spec now passes, including welcome, navigation, staging, untracked, and deleted-file transitions.

Coarse projections now use nearest-line rounding rather than always biasing toward the start or end of a changed span. The expanded overlapping-conflict fixture retained equal prefixes at every recorded anchor in center and bottom layouts, and focused pure-model tests cover expanded, contracted, overlapping, inner-mapped, source-composed, and moved ranges. The result mapping now composes exact unchanged regions from either input while retaining the direct base mapping as fallback. Accept incoming/current/both in both concatenation orders, discard, a manual insertion inside an accepted block, undo, and restoration all retained equal prefixes.

### Current complexity and performance

- Building and measuring the plan is approximately O(number of alignment intervals), currently close to O(lines) for unchanged regions.
- Each synchronized scroll first uses guarded bounded `max_height` lookup. Proven text endpoints take O(1) renderer calls; ambiguous text, fold, or filler endpoints retain O(log lines + log virtual columns) measured fallback. Scrolling never rebuilds padding.
- Conflict result edits compute three merge-base mappings plus incoming-to-result and current-to-result mappings, compose exact regional source mappings, and rebuild all three panes' padding.
- Final Neovim 0.12.2 cold/warm alignment measured approximately 8/2 ms for 1k lines, 72/14 ms for 10k lines, and 775/75 ms for 100k lines. Nightly measured 8/1 ms, 73/15 ms, and 727/78 ms.
- The 100k warm run reused 9,999 padding extmarks and updated only the changed interval. An earlier cold phase profile measured about 187 ms for plan building, 290 ms for redraw/measurement, and 273 ms for padding installation; warm reuse reduced the corresponding work substantially.
- Measurement/padding and view synchronization are isolated from `./lua/codediff/ui/wrap_alignment.lua`; the coordinator retains conflict and lifecycle orchestration.

### Measurement cache design

The first cache is window-owned because measured height depends on the pane, not only the buffer. Each pane state retains heights only for its current rendering signature. The signature includes window width, effective gutter width, wrapping and continuation options, number/sign/fold/status columns, tab width, and ambiguous-width policy.

Cache entries use exact line text rather than buffer `changedtick`. A changed tick would invalidate every interval after any edit and prevent unchanged-line reuse. Reuse is restricted to semantic intervals that are:

- One real line in every pane.
- Textually equal across every pane.
- Free of move-leading rows, virtual lines, virtual text, and conceal extmarks.
- Outside compact mode and windows with active conceal.

This deliberately excludes changed, inserted, deleted, folded, decorated, and multiline intervals. Those continue to call `nvim_win_text_height()`. Buffer replacement clears pane state; render-signature changes clear that pane's cache. Exact content identity naturally invalidates edited lines while preserving unchanged lines across changed ticks and shifted row numbers.

Instrumentation is exposed through each alignment result and `wrap_alignment.get_metrics()`: rebuild count and reason, interval count, measurement requests, actual measurements, cache hits, and elapsed time for plan construction, preparation, fold scanning, redraw/measurement, padding installation, and view synchronization. `wrap_alignment.reset_metrics()` provides a stable counter for interactive checks that scrolling does not rebuild alignment.

Window-option ownership is tracked independently from padding state. The first policy application captures `wrap`, `scrollbind`, `smoothscroll`, and `cursorbind` per tabpage and logical side. If any aligned pane had smooth scrolling enabled, the active alignment group enables it consistently across every pane; cleanup still restores each pane's original value. Temporary clears restore but retain the profile so side/inline/side window replacement cannot overwrite the original values. Session release restores every surviving owned window and removes the profiles. Direct side-by-side, direct inline, result panes, layout transitions, single-pane cleanup, wrapped mode, and `wrap=false` all use the same ownership API.

The cache does not attempt to reuse arbitrary external decoration state. Supported height-affecting extmarks bypass reuse on their rows; unobservable decoration-provider changes still require explicit invalidation or documentation.

Two-pane semantic plans are cached per alignment window and buffer pair. Reuse requires matching original and modified buffer identities and changed ticks plus a structural fingerprint of the diff's changes and moves. The fingerprint allows an equivalent freshly computed diff result to reuse the plan without trusting object identity. Replacing buffers or clearing an alignment group replaces or clears its bounded cache entry. Conflict plans are not cached yet because result refresh currently recomputes all base/input mappings before plan construction.

Padding is preserved only when a two-pane plan cache hit proves the interval topology and buffer contents are unchanged and compact mode is inactive. Intervals whose natural heights miss the height cache have their old padding removed before measurement; unchanged intervals keep their extmarks. Rendering then reuses extmarks whose desired filler count is unchanged and updates only changed counts. Plan misses, edits, compact folds, and conflict renders retain the full-clear fallback.

Padding metrics report reused, updated, and removed extmarks. A later tier may preserve padding across plan changes, but only after padding/decorations acquire explicit generations and edit dirty ranges.

## Product decisions

1. Is Neovim 0.12+ acceptable for aligned multi-pane wrapping, with complete inline wrapping gated to 0.13+?
2. Is `diff.wrap` a boolean, with `true` guaranteeing aligned wrapping?
3. CodeDiff temporarily owns `wrap`, `scrollbind`, `smoothscroll`, and `cursorbind` while aligned wrapping is active and restores the captured user values on policy transitions and cleanup.
4. Which external height-affecting decorations should be supported or suppressed?
5. Must arbitrary external fold commands remain supported, or can compact folds be CodeDiff-owned?
6. Should all modes ship together, or should complete two-pane wrapping land before conflict and inline support?
7. What performance budget is acceptable for initial render, edit refresh, and resize?
8. Should an explicit `alignment.invalidate(tabpage)` API be exposed for integrations?

## Non-goals

- No timer-based reapplication of `wrap`.
- No per-scroll full-document recomputation.
- No hand-written clone of Neovim's wrapping algorithm.
- No global `diffexpr` mutation.
- No silent independent-scrolling fallback presented as correct alignment.
- No unrelated `signcolumn` preference changes.
- No dedicated mirror buffers unless window-scoped namespace measurement fails.

## Suggested maintainer-facing proposal

### Title

`feat(ui): align wrapped diff panes by measured display rows`

### Description

Add correctly synchronized wrapping using semantic alignment intervals, `nvim_win_text_height()`, and window-scoped filler namespaces. Each pane receives width-specific virtual-row compensation, while lightweight semantic view synchronization replaces native `scrollbind` in all supported multi-pane views. On Neovim 0.13, inline deleted lines use wrapped virtual-line overflow. The implementation avoids timer workarounds, arithmetic wrap estimation, and per-scroll padding recomputation.
